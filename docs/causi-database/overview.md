# Banco de Dados — Visao Geral

O Causi utiliza **PostgreSQL** hospedado no **Supabase**, organizado em **3 schemas** principais e um bucket de storage.

---

## Schemas

| Schema | Propósito | Tabelas | Documentação |
|--------|-----------|---------|-------------|
| `public` | Dados operacionais: contas, usuários, deals, contatos, conversas, mensagens, tarefas, agentes | 24 | [schema-public.md](./schema-public.md) |
| `billing` | SaaS billing: planos, períodos, addons, assinaturas, pagamentos | 7 | [schema-billing.md](./schema-billing.md) |
| `classroom` | Educação: cursos, módulos, aulas, certificados, progresso | 7 | [schema-classroom.md](./schema-classroom.md) |

Além dos schemas, o Supabase Storage é utilizado para arquivos — veja [storage.md](./storage.md).

## Convenções de Nomes

- **IDs**: Todas as tabelas usam `bigint GENERATED BY DEFAULT AS IDENTITY` como PK, exceto `users` (uuid, espelhando `auth.users`) e `user_certificates` (uuid via `gen_random_uuid()`).
- **Timestamps**: Todas as tabelas possuem `created_at` e `updated_at` com default `now()`.
- **Foreign keys**: Nomeadas como `<tabela>_<coluna>_fkey` (ex: `deals_account_id_fkey`).
- **Isolamento multi-tenant**: Quase todas as tabelas operacionais possuem `account_id` (direto ou via entidade pai). RLS utiliza `account_id` como base de isolamento.
- **Slugs**: `accounts`, `users`, `pipelines`, `plans`, `plan_periods` e `addons` possuem `slug` único gerado automaticamente via trigger.
- **Soft deletes**: Não utilizamos soft delete global. Tabelas como `deals` e `channels` possuem `is_active` para controlar visibilidade (é visível mas bloqueado).

| Recurso | Uso |

| **Supabase Realtime** | Notificações em tempo real (novas mensagens, atualizações de deals) |

## Supabase CLI — Workflow de Desenvolvimento

> **Regra**: Alterações estruturais (tabelas, colunas, indexes, enums) devem ser feitas **exclusivamente** via Supabase CLI e registradas em migrations. **Nunca** use o SQL Editor diretamente para alterar o schema em produção.

### Setup Inicial

```bash
# 1. Instalar dependências (CLI incluso no projeto)
pnpm install

# 2. Login no Supabase
supabase login

# 3. Vincular ao projeto remoto
supabase link --project-ref <dev-project-ref>

# 4. Baixar schema remoto
supabase db pull
```

> O comando `supabase db pull` requer Docker Desktop em execução, pois inicia um container Postgres local para comparar schemas.

### Comandos Frequentes

| Comando | Descrição |
|---------|-----------|
| `supabase login` | Autenticar no Supabase (gera token) |
| `supabase link --project-ref <dev-project-ref>` | Vincular pasta local ao projeto remoto |
| `supabase db pull` | Baixar schema remoto → `supabase/migrations/` |
| `supabase gen types typescript > src/lib/database.types.ts` | Gerar tipagem TypeScript em `src/lib/database.types.ts` |
| `supabase db diff -f <nome>` | Gerar migration com diff entre estado local e remoto |
| `supabase migration new <nome>` | Criar nova migration vazia |
| `supabase db seed` | Executar `supabase/seed.sql` |

### Seeds

O arquivo `supabase/seed.sql` contém dados de configuração (planos, roles). Seeds são **idempotentes** (`INSERT ... ON CONFLICT DO UPDATE`) e atuam apenas em tabelas de configuração — nunca em dados operacionais.

## Estrutura de Diretórios do Schema

```
supabase/
├── migrations/           # Arquivos SQL versionados (YYYYMMDDHHMMSS_nome.sql)
├── schemas/              # Organização lógica do schema por tipo
│   ├── config.sql        # Configurações gerais
│   ├── enums.sql         # Definições de enums
│   ├── public/
│   │   ├── tables/       # Definições de tabelas
│   │   ├── functions/    # Funções SQL (RPC, helpers)
│   │   ├── triggers/     # Triggers e funções de trigger
│   │   ├── rls/          # Políticas de Row-Level Security
│   │   ├── views/        # Views SQL
│   │   ├── constraints/  # Constraints adicionais
│   │   └── indexes/      # Índices
│   ├── billing/          # (mesma estrutura)
│   ├── classroom/        # (mesma estrutura)
│   └── storage/
│       └── rls/          # Políticas RLS do Storage
├── functions/            # Edge Functions (TypeScript/Deno)
├── seed.sql              # Dados iniciais idempotentes
└── config.toml           # Configuração do CLI
```

## Fluxo Declarativo de Schemas

O projeto utiliza arquivos SQL declarativos em `supabase/schemas/` como **fonte de verdade** para o schema do banco de dados. As migrations em `supabase/migrations/` são geradas automaticamente a partir desses arquivos.

### Workflow de Alteração

1. **Editar** os arquivos `.sql` em `supabase/schemas/<schema>/<tipo>/` (ex: `supabase/schemas/public/tables/deals.sql`)
2. **Gerar migration** com `supabase db diff -f <nome_descritivo>` — compara os schema files contra uma shadow database local (requer Docker). **Não usar `--linked`** — isso mudaria o target para o banco remoto, quebrando o workflow declarativo.
3. **Revisar** a migration gerada em `supabase/migrations/<timestamp>_<nome>.sql` — verificar se tudo foi capturado. RLS policies (alter), DML, view ownership e outras entidades com caveats podem precisar de ajuste manual (veja a lista completa em `.agents/rules/declarative-database-schema.md`).
4. **Aplicar** com `supabase db push` (recomendado `--dry-run` antes)

> **Nunca** edite migrations diretamente — elas são geradas pelo CLI a partir dos schemas declarativos, exceto nos casos de caveats onde é necessário criar uma migration vazia via `supabase migration new`.

## Ambientes

O projeto possui **dois projetos Supabase** separados no Cloud:

| Projeto | Uso | Branch Git | Configuração |
|---------|-----|-----------|-------------|
| `causi-dev` | Desenvolvimento e testes | Todas (exceto `main`) | Chaves em `.env.local` |
| `causi-prod` | Produção | `main` | Chaves via Vercel Environment Variables |

> **Nunca** aplique migrações diretamente no projeto de produção sem antes testar no projeto de desenvolvimento.

### Trocando de Ambiente

```bash
# Vincular ao projeto de desenvolvimento
supabase link --project-ref <dev-project-ref>

# Vincular ao projeto de produção (apenas na branch main)
supabase link --project-ref <prod-project-ref>
```

O arquivo `supabase/.temp/project-ref` armazena o ID do projeto atualmente linkado. Confirme o target antes de um `db push` com `supabase migration list`.

> **Guia operacional completo** (merge de branches, conflitos em migrations, alternância `.env` / `.env.test`, checklist pós-merge): [guides/supabase-migrations-and-environments.md](../guides/supabase-migrations-and-environments.md)

### Workflow Completo (Development → Production)

```bash
# 1. Editar schemas → supabase/schemas/<schema>/<tipo>/<arquivo>.sql

# 2. Gerar migration (requer Docker)
supabase db diff -f <nome_descritivo>

# 3. Revisar a migration gerada (⚠️ verificar caveats)

# 4. Linkar ao projeto develop e aplicar
supabase link --project-ref <dev-project-ref>
supabase db push --dry-run  # visualizar antes
supabase db push

# 5. Gerar tipos TypeScript atualizados
supabase gen types typescript > src/lib/database.types.ts

# --- Após testes no projeto dev, deploy para produção ---

# 6. Trocar para o projeto de produção (branch main)
supabase link --project-ref <prod-project-ref>

# 7. Aplicar as mesmas migrations
supabase db push
```

## Acesso a Dados: Supabase SDK

O projeto usa **exclusivamente o Supabase SDK** (`@supabase/supabase-js` + `@supabase/ssr`). ORMs como Prisma/Drizzle **não são permitidos** — eles bypassam RLS.

| Contexto | Client | Chave |
|----------|--------|-------|
| Client Components | `createBrowserClient` via `@supabase/ssr` | `anon` (pública) |
| Server Components / Actions | `createServerClient` via `@supabase/ssr` | `anon` + cookies |
| Tarefas admin (bypassa RLS) | `createClient` via `@supabase/supabase-js` | `SERVICE_ROLE_KEY` (nunca no client) |

Arquivos de client:
- `lib/supabase/client.ts` — Client Components
- `lib/supabase/server.ts` — Server Components / Server Actions

## Troubleshooting

### Role inexistente ao rodar `db pull`

Se o CLI reportar erro como `role "readonly_ai" does not exist`, crie o role no SQL Editor:
```sql
CREATE ROLE readonly_ai NOLOGIN;
```

### Histórico de migrations inexistente

Se o banco foi criado/modificado via SQL Editor da UI antes de adotar o CLI, a tabela `supabase_migrations.schema_migrations` pode não existir. Use o comando de reparo:
```bash
supabase migration repair --status applied <migration_id>
```
Este comando apenas ajusta a tabela técnica de controle — não altera dados de negócio.

### Boas práticas pós-setup

1. **Schema via CLI**: Todas as mudanças estruturais via `supabase migration new` ou `supabase db diff -f`.
2. **Consultas via SDK**: Utilize o Supabase SDK conforme documentado na [ADR](../decisions/adr.md).
3. **SQL Editor apenas para consultas**: Use o editor da UI apenas para visualização e testes rápidos, nunca para alterar schema em produção.
